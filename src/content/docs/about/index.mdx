# О проекте

## Что такое @vue-modeler/model

Библиотека для управления состоянием на основе моделей для [Vue.js](https://vuejs.org/). Крайне простой API служит единственной цели — созданию моделей. Сохраняет типизацию, поддерживает принципы ООП, DRY и SOLID.

---

## Ключевые особенности

- **Нет глобального хранилища состояния**. Нет хранилища — нет проблем. Состояние инкапсулировано в модели и является реактивным. Вне модели оно доступно только для чтения и наблюдения.
- **Крайне простой API**: Прототип модели — это стандартный класс, действия определяются как стандартные методы с декораторами. Создайте модель, вызвав фабричный метод — больше ничего не нужно. Реактивные свойства и поведение определяются с помощью стандартного Vue API.
- **Поддерживает общие модели**. Встроенный контейнер моделей гарантирует, что модель остается в памяти, пока используется. Неиспользуемые модели автоматически удаляются из контейнера.
- **Следует принципу DRY**. Действия имеют собственное состояние. Нет необходимости писать дополнительный код для отслеживания состояний действий. Вместо этого сосредоточьтесь на проектировании архитектуры и бизнес-логики.
- **Придерживается ООП**. Прототипы моделей — это классы, поддерживающие наследование, инкапсуляцию, полиморфизм и деструкторы.
- **Соответствует принципам SOLID**. Поощряет чистые практики кодирования, такие как композиция моделей, внедрение зависимостей через конструкторы, разделение интерфейсов, единственная ответственность и т.д.
- **Сохраняет типобезопасность**. Все подсказки автодополнения будут работать как внутри, так и вне контекста класса.

## Цели создания

### **Устранение шаблонного кода в действиях и сокращение кодовой базы**

#### Проблемы:
- Действия часто сопровождаются повторяющимся кодом для отслеживания выполнения через дополнительные переменные, такие как `isLoading`, `isPending`. Этот код добавляет мало ценности к бизнес-логике, но раздувает кодовую базу.
- Отмена или блокировка действия может быть сложной. Разработчики склонны изобретать велосипед, даже несмотря на то, что эти операции являются общими паттернами.
- Обработка исключений вносит дополнительные осложнения — разработчики либо забывают о ней, либо обрабатывают её непоследовательно без четких руководств.

#### Решения:
- Действие представлено как объект с реактивными свойствами, отражающими статус его выполнения: `ready`, `pending`, `lock`, `abort`, `error`.
- Легко блокируйте или отменяйте действие с помощью методов `lock` и `abort`.
- Действия перехватывают исключения, создают ошибки, сохраняют их как часть состояния и предоставляют унифицированный интерфейс для обработки.

Меньше кода — меньше проблем.

### **Использование классов и интерфейсов**

И VUEX, и Pinia используют кастомные подходы для управления хранилищами и повторного использования общего кода. Я хотел использовать стандартные классы, интерфейсы, геттеры и защищенные свойства. Поэтому мои модели построены на классах и поддерживают паттерны ООП.

### **Повторное использование бизнес-логики между проектами**

Модели — это классы, где зависимости внедряются через конструкторы, поэтому файлы моделей не содержат прямых импортов внешних зависимостей. Модели легко извлекаются в отдельные модули и повторно используются в других проектах или с разными UI библиотеками.

## Дополнительные преимущества

- **Многослойная структура проекта**:
  - Слой приложения содержит определения моделей.
  - Инфраструктурный слой включает фабрики запросов/ответов и API вызовы.
  - UI слой обрабатывает шаблоны, фильтры и логику отображения.
  - Конфигурационный слой — это контейнер зависимостей, где разработчики устанавливают отношения между моделями и другими слоями.
- **Четкая структура папок**. Каждый слой имеет свою выделенную директорию. Внутри директорий приложения и инфраструктуры подпапки организованы по доменным областям.

---

Эта библиотека обещает упростить управление состоянием в приложениях [Vue.js](https://vuejs.org/), одновременно способствуя чистому, масштабируемому и поддерживаемому коду.
